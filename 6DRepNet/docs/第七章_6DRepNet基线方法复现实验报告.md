# 七、6DRepNet 基线方法复现实验

## 7.1 实验目标与复现范围

在前文完成头部姿态估计任务调研和 6DRepNet 论文学习之后，本章节进一步围绕课程实验要求中的“基线方法复现”展开。复现工作的目标不是只让代码能够运行，而是尽可能还原作者公开代码库中的真实实验流程：一方面跑通作者提供的实时 demo，验证模型在普通摄像头视频流上的实际推理效果；另一方面在作者论文使用的公开测试集 AFLW2000 上运行评估代码，并将误差结果与论文表格进行逐项对照，从而判断复现是否真正可信。

本次复现最终采用的权重文件为 `6DRepNet/checkpoints/6DRepNet_300W_LP_AFLW2000.pth`。该权重对应论文 Table 1 中在 AFLW2000 数据集上报告的 6DRepNet 结果，也是本次复现实验的核心基线。由于 BIWI 数据集官方下载入口已经失效，并且课程助教明确说明只复现 AFLW2000 一个数据集即可，因此本节不再展开 BIWI 数据集实验，而是集中保证 AFLW2000 流程的正确性与结果的一致性。

## 7.2 项目搭建与运行记录

### 7.2.1 代码库与实验环境

6DRepNet 作者提供了完整的公开代码库，其中包含网络结构、训练脚本、测试脚本、摄像头 demo 以及预训练模型说明。本实验首先将项目代码整理到本地工作目录 `6DRepNet/` 下，并在其基础上补充了用于课程复现的脚本目录 `6DRepNet/script/`。这样做的原因是：原始仓库中的命令主要面向作者自己的目录结构和实验习惯，直接运行时容易出现路径、数据列表和环境依赖不一致的问题；而课程复现需要留下清晰的、可重复执行的运行记录，因此将 demo 和评估流程分别封装为独立脚本。

实验主要在本地 WSL Ubuntu 环境中完成，使用 Python 虚拟环境管理依赖，并在本地 NVIDIA RTX 4060 GPU 上运行推理与评估。项目依赖主要包括 PyTorch、torchvision、OpenCV、matplotlib、NumPy、SciPy、pandas 以及用于实时人脸检测的 `face_detection` 包。基本环境配置流程如下：

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip setuptools wheel
pip install -r 6DRepNet/requirements.txt
pip install "git+https://github.com/elliottzheng/face-detection.git@master"
```

由于本次需要运行摄像头 demo，还额外安装了与图形界面和摄像头访问有关的系统依赖。例如，`matplotlib` 的 `TkAgg` 后端需要 `python3-tk`，OpenCV 的窗口显示依赖 Qt/X11 相关库；在 WSL 中访问 Windows 摄像头时，还需要将摄像头设备正确挂载到 `/dev/video0`，并确保当前用户属于 `video` 用户组。

### 7.2.3 运行脚本设计

为了让 demo 和 AFLW2000 测试能够一键复现，本实验在 `6DRepNet/script/` 下整理了两个主要脚本。

第一个脚本是 `run_demo.sh`，用于运行 demo 程序。它支持两种模式：一种是使用静态图像快速验证模型与权重是否能正常推理；另一种是使用摄像头实时读取视频流，调用人脸检测器定位人脸，再将裁剪后的人脸送入 6DRepNet 得到 yaw、pitch、roll，并以三维立方体的形式叠加显示头部姿态。摄像头 demo 的运行命令为：

```bash
DEMO_MODE=camera CAM_ID=0 bash 6DRepNet/script/run_demo.sh
```

第二个脚本是 `run_eval_aflw2000.sh`，用于在 AFLW2000 上运行测试。脚本会检查 checkpoint、数据目录和文件列表是否存在，并调用辅助测试程序完成批量推理、误差统计和日志保存。最终的运行命令为：

```bash
bash 6DRepNet/script/run_eval_aflw2000.sh
```

这种脚本化的组织方式使复现实验具有较好的可追踪性：每一次运行都会在输出目录中留下日志，便于后续将命令、环境、结果和论文表格进行对应。

### 7.2.4 遇到的问题与解决方式

本次复现过程中遇到的问题并不只是“缺少某个包”这么简单，而是覆盖了数据集获取、Python 版本兼容、WSL 摄像头访问、OpenCV 图形界面、人脸检测器调用和评估协议一致性等多个层面。这些问题本身也反映了论文复现中的一个重要事实：算法论文中的核心思想固然重要，但真正复现出可信结果，往往还需要把工程细节逐一对齐。

首先遇到的是 BIWI 数据集下载失败问题。原先文档中给出的 BIWI 官方链接已经无法访问，网页返回 404。经过与课程要求对照，并结合助教在课程群中的说明，本实验将复现范围调整为 AFLW2000 一个数据集。这样既符合课程要求，也避免将主要时间消耗在已经失效的数据下载入口上。

第二个问题是依赖安装路径。最初在项目根目录之外直接执行 `pip install -r requirements.txt` 时，终端提示找不到 `requirements.txt`。原因是当前 shell 所在目录是总工程目录，而依赖文件位于 `6DRepNet/requirements.txt`。解决方法是显式使用相对路径：

第二个问题来自 Python 与 NumPy 版本兼容。原代码中存在对 NumPy 私有接口的导入，例如 `numpy.lib.function_base` 下的内部函数。新版本 NumPy 中这些私有接口并不稳定，因此会出现 `ModuleNotFoundError`。该导入并不是 6DRepNet 推理逻辑真正需要的部分，因此通过删除无用私有导入，保证代码只依赖稳定公开 API。

第三个问题出现在摄像头 demo 的图形界面部分。原始 `demo.py` 使用 `matplotlib.use('TkAgg')`，如果系统缺少 `tkinter`，会报错 `No module named 'tkinter'`。该问题属于系统依赖缺失，可以通过安装 `python3-tk` 解决。随后又遇到 OpenCV Qt 插件 `xcb` 无法初始化的问题，这是 Linux/WSL 图形库依赖不完整导致的，需要补充安装 `libxcb-xinerama0`、`libxcb-cursor0`、`libxkbcommon-x11-0` 等 X11/Qt 相关库。完成这些配置后，OpenCV 才能正常弹出实时显示窗口。

第四个问题是 WSL 中摄像头权限与格式。Windows 摄像头通过 USB/IP 挂载到 WSL 后，设备显示为 `/dev/video0` 和 `/dev/video1`，其中 `/dev/video0` 是真正的视频采集设备，`/dev/video1` 主要是元数据设备。最初直接运行 OpenCV 会出现 `Cannot open webcam` 或 `Permission denied`，需要将当前用户加入 `video` 组，并通过 `newgrp video` 让权限立即生效。之后使用 `v4l2-ctl` 检查摄像头格式，发现摄像头支持 MJPG 和 YUYV，其中 MJPG 在 640×480 下可以稳定输出。于是摄像头 demo 中显式设置 V4L2 后端、MJPG 编码和合适分辨率，避免默认格式导致的读取超时。

第五个问题是 `face_detection` 包的检测器调用。运行实时 demo 时，视频帧首先由 OpenCV 读取，然后送入 RetinaFace 类检测器定位人脸。如果安装到的是不完整或不匹配的包版本，可能会触发 `NotImplementedError`。解决方法是安装作者 demo 所需的 `face_detection` GitHub 版本，并确认实际调用的是可用的 RetinaFace 检测器实现。

最后，也是最影响复现结论的问题，是 AFLW2000 测试结果与论文表格一开始不一致。初始测试使用了全部 2000 个样本，得到的 MAE 约为 4.86，高于论文中 3.97 的结果。进一步检查作者测试逻辑后发现，原始评估流程会过滤掉角度绝对值超过 99° 的样本，实际测试样本数为 1969。重新按照作者协议生成 `files_filtered_99.txt` 后，测试结果与论文 Table 1 完全对齐。这说明此前误差偏高不是 checkpoint 错误，也不是模型推理错误，而是测试样本列表与论文协议不一致导致的。


该视频长度约 121 秒，分辨率为 2416×1440，记录了模型在本地摄像头输入下进行实时头部姿态估计的过程。从整体效果看，demo 能够完成完整的“视频采集—人脸检测—姿态估计—结果可视化”流程：当人脸被检测到后，系统会在人脸区域上叠加一个三维立方体，立方体的朝向随头部 yaw、pitch、roll 的变化而变化。因此，demo 不只是输出三个角度数值，而是把抽象姿态估计结果转化为直观的空间方向显示。

图7-1 摄像头 demo 初始帧：正脸附近姿态下，模型能够稳定检测人脸并绘制头部姿态立方体。

从视频开头可以看到，在人脸距离摄像头较近、面部轮廓完整的情况下，模型能够较稳定地输出姿态。即使背景中存在窗户和较强背光，系统仍然可以检测到人脸，并绘制出大致符合头部朝向的立方体。这说明 6DRepNet 在常规近距离场景下具有较好的可用性：RetinaFace 检测器负责提供较准确的人脸框，6DRepNet 则基于裁剪后的人脸图像直接回归旋转表示。相比需要精确关键点的传统 PnP 方法，这种 landmark-free 的深度回归方式对局部关键点误差不那么敏感，因此在普通摄像头画面中更容易跑通。

不过，视频中也暴露出实时系统的第一个明显短板：光照与距离会显著影响检测稳定性。

[run_demo.mp4  00:12]

图7-2 光照较暗或人脸有效区域不足时，检测器可能短暂漏检，导致姿态立方体消失。

在部分帧中，人脸虽然对人眼仍然可见，但立方体会短暂消失。这通常不是 6DRepNet 主干网络本身没有输出，而是前置人脸检测器没有给出满足置信度阈值的人脸框。6DRepNet 的推理前提是已经获得可靠的人脸 crop；一旦检测器漏检，后续姿态网络就没有输入。因此，整个 demo 的稳定性实际上由“检测器稳定性”和“姿态回归稳定性”共同决定。背光、低照度、脸部面积较小、运动模糊都会降低检测器置信度，从而造成输出中断。这也提示我们，在真实应用中不能只评价姿态网络在标准数据集上的 MAE，还必须考虑上游检测模块在真实场景中的鲁棒性。

视频中第二个值得注意的现象是姿态立方体会出现轻微抖动。

[run_demo.mp4  00:30]

图7-3 手持手机或局部遮挡场景：模型仍可估计姿态，但遮挡会增加检测框和姿态输出的不稳定性。

当头部缓慢移动或画面中出现手机遮挡时，立方体总体方向仍然能够跟随头部变化，但在相邻帧之间可能出现小幅跳动。这种抖动的来源主要有三点。第一，人脸检测框本身在每一帧都是独立预测的，框的位置和大小会有像素级波动；裁剪区域一旦变化，送入 6DRepNet 的人脸外观也会变化。第二，6DRepNet 原始 demo 是单帧推理，没有使用卡尔曼滤波、指数滑动平均或时序网络对输出角度进行平滑，因此相邻帧之间的微小预测误差会直接表现为可视化立方体的抖动。第三，遮挡会破坏面部局部纹理，例如嘴部、鼻梁或下颌区域被遮住后，网络可用的姿态线索减少，预测自然会更依赖剩余区域和训练集先验。

第三个现象是多脸或误检场景下的目标选择问题。

[run_demo.mp4  00:45]

图7-4 画面中出现手机屏幕上的人脸时，系统可能同时对真实人脸和屏幕人脸绘制姿态。

在视频中，当手机屏幕里也出现人脸图像时，demo 可能会同时在真实人脸和屏幕中的人脸上绘制立方体。这一现象很有启发性：当前系统只是对图像中所有满足阈值的人脸区域进行检测和姿态估计，并不知道哪一个是实验者真正关心的目标。换言之，6DRepNet demo 解决的是“每个检测到的人脸朝向哪里”，而不是“持续跟踪同一个特定用户的头部朝向”。在多人会议、课堂监控或手机屏幕干扰场景中，如果没有额外的身份跟踪、目标选择或时序关联模块，系统就可能对非目标人脸也输出结果。

总体来看，demo 的优点非常明确。首先，它的流程完整，能在普通摄像头画面中实时完成头部姿态估计，说明模型具有较好的工程可运行性。其次，三维立方体可视化非常直观，可以让观察者立刻理解 yaw、pitch、roll 的变化方向。再次，6DRepNet 使用连续 6D 旋转表示回归旋转矩阵，再转换为欧拉角展示，相比直接回归欧拉角更符合三维旋转空间的几何结构，因此在常规姿态范围内表现较稳定。

与此同时，demo 也揭示了该方法在真实应用中的局限。其一，系统高度依赖人脸检测器，检测器漏检会导致姿态估计完全中断。其二，原始 demo 缺少时序平滑机制，所以立方体在连续帧中可能出现抖动。其三，模型训练主要依赖 300W-LP 等人脸姿态数据，真实摄像头中的背光、低照度、屏幕反射、遮挡和低分辨率人脸会带来域偏移。其四，多脸场景下缺少目标跟踪和身份绑定，容易对非目标人脸也进行估计。由此可见，6DRepNet 的核心姿态回归能力较强，但若要部署到更复杂的真实场景，还需要在检测、跟踪、光照增强和时序稳定性方面继续补强。

## 7.4 AFLW2000 数据集测试结果与分析

### 7.4.1 测试设置

AFLW2000 是头部姿态估计任务中常用的测试集之一，包含 2000 张具有三维姿态标注的人脸图像。本实验使用作者论文 Table 1 对应的 checkpoint `6DRepNet_300W_LP_AFLW2000.pth` 进行测试。为了与论文评估协议保持一致，测试列表采用过滤后的 `files_filtered_99.txt`，共 1969 个样本。

测试时，图像首先按照作者代码中的预处理方式进行缩放、中心裁剪和 ImageNet 标准化，然后送入 6DRepNet 网络。网络输出 6D 旋转表示，经 Gram-Schmidt 正交化转换为 3×3 旋转矩阵，再由旋转矩阵转换为 yaw、pitch、roll 三个欧拉角。最后分别计算三个角度的平均绝对误差，并取三者均值作为总 MAE。

本次实验的原始日志如下：

```text

Device: cuda:0
Samples: 1969
Yaw: 3.6258, Pitch: 4.9073, Roll: 3.3737, MAE: 3.9689
Paper reference on AFLW2000: Yaw 3.63, Pitch 4.91, Roll 3.37, MAE 3.97
```

将复现结果与论文 Table 1 中的 AFLW2000 结果对比如下：

| 来源 | Yaw MAE | Pitch MAE | Roll MAE | Mean MAE |
| --- | ---: | ---: | ---: | ---: |
| 论文 Table 1 | 3.63 | 4.91 | 3.37 | 3.97 |
| 本次复现 | 3.6258 | 4.9073 | 3.3737 | 3.9689 |

可以看到，本次复现的 yaw、pitch、roll 以及三者平均 MAE 都与论文结果高度一致，差异仅来自小数位保留方式。特别是平均误差 3.9689 与论文报告的 3.97 基本完全相同，说明本实验已经成功复现了论文 Table 1 中 6DRepNet 在 AFLW2000 上的测试结果。

### 7.4.2 模型优势分析

从 AFLW2000 的定量结果看，6DRepNet 在三个角度上的误差都较低，平均 MAE 约为 3.97°。这说明模型能够从单张人脸图像中较准确地恢复三维头部朝向。结合前文的论文学习可以看到，其优势主要来自三个方面。

第一，6DRepNet 没有直接将 yaw、pitch、roll 当作三个互相独立的标量回归，而是先预测连续 6D 旋转表示，再构造合法旋转矩阵。这使网络输出更接近真实的 SO(3) 旋转空间，减少了欧拉角不连续和多解性带来的学习困难。也就是说，模型训练时面对的是更平滑、更适合神经网络拟合的旋转表示；而最终展示或评价时，再转换回人类容易理解的欧拉角。

第二，模型采用 RepVGG 风格的卷积骨干，在训练时具有较强表达能力，在部署时又可以转换为推理友好的结构。对于头部姿态估计这种需要实时性的任务来说，网络不仅要准确，还要能在普通 GPU 或甚至边缘设备上保持足够快的推理速度。本次摄像头 demo 能够实时运行，也从侧面验证了这种结构设计的工程价值。

第三，6DRepNet 不依赖显式人脸关键点。传统基于 PnP 的方法需要先检测眼角、鼻尖、嘴角等关键点，再通过三维头模求解姿态；一旦关键点在侧脸、遮挡或模糊场景下不准，姿态结果会明显退化。6DRepNet 直接从整个人脸外观中学习姿态线索，可以利用更丰富的纹理和轮廓信息，因此在 AFLW2000 这类非严格受控数据集上具有较强表现。

### 7.4.3 模型局限性分析

尽管 AFLW2000 上的复现结果非常接近论文，仍然不能简单认为 6DRepNet 已经解决了所有真实场景问题。首先，从误差分布看，pitch 误差 4.9073° 高于 yaw 和 roll。这在头部姿态估计中较常见，因为俯仰角变化往往伴随更多透视形变和遮挡：低头时眼睛、眉骨、鼻梁阴影变化明显；抬头时下颌和鼻孔区域更加突出。与左右转头相比，pitch 的视觉线索更容易受到相机高度、脸部比例和光照方向影响。

其次，AFLW2000 是静态图像测试集，其评价指标只关心单帧角度误差，并不评价视频连续性。因此，一个模型即使在 AFLW2000 上 MAE 很低，也可能在实时 demo 中出现帧间抖动。换言之，AFLW2000 证明的是“单张图像姿态回归准确”，而 demo 进一步暴露的是“连续视频输出是否稳定”。这两者不是同一个问题。

再次，AFLW2000 测试使用的是已经裁剪并标注好的人脸数据，而真实摄像头 demo 需要先检测人脸。这样一来，真实系统的误差不仅来自 6DRepNet，也来自人脸检测框的位置、大小和置信度。若检测框偏移，模型看到的人脸 crop 就会发生变化；若检测器漏检，则姿态估计无法进行。因此，真实部署中的性能下限往往由整个视觉管线中最脆弱的环节决定。

最后，6DRepNet 的输出虽然通过 6D 表示保证了旋转矩阵的连续性和合法性，但最终评价仍然转换为欧拉角 MAE。欧拉角本身仍具有周期性和坐标约定问题，因此在极端姿态或接近角度边界的样本上，误差解释需要谨慎。作者通过 99° 过滤协议排除了部分极端样本，这有助于公平比较主流姿态范围内的性能，但也说明模型在极端姿态下的表现仍需要单独评估。

## 7.5 实验总结与收获

通过本次复现实验，可以确认 6DRepNet 的公开代码和预训练权重具有较好的可复现性。在正确安装依赖、准备 AFLW2000 数据集、使用作者过滤协议并加载 `6DRepNet_300W_LP_AFLW2000.pth` 后，本实验得到的 AFLW2000 测试结果与论文 Table 1 完全一致，平均 MAE 为 3.9689°。这说明论文中的核心定量结论可以在本地环境中稳定复现。

同时，摄像头 demo 的运行也让我们更直观地理解了 6DRepNet 的实际能力。它能够在普通摄像头输入中实时估计头部姿态，并用三维立方体展示方向变化，说明模型不仅停留在数据集指标上，也具备较强的工程演示价值。但 demo 中出现的漏检、抖动、多脸干扰和光照敏感等现象也提醒我们：一个完整的头部姿态估计系统并不只是一个姿态回归网络，还包括人脸检测、图像预处理、目标跟踪、时序平滑和可视化等多个环节。任何一个环节不稳定，最终用户看到的效果都会受到影响。

从方法角度看，6DRepNet 最值得学习的地方在于它没有盲目堆叠网络结构，而是从旋转表示这一根本问题入手，将不连续的欧拉角回归转化为连续 6D 旋转表示学习。这种思路体现了几何先验与深度学习的结合：网络负责从图像中学习复杂外观特征，旋转表示则负责让输出空间更符合三维几何规律。对于后续创新设计而言，本实验也给出了明确方向：若要进一步提升真实视频场景中的表现，可以考虑在人脸检测后加入目标跟踪，在姿态输出后加入时序滤波，或者引入低照度增强与遮挡鲁棒训练，以弥补当前 demo 中暴露出的工程短板。

综上，我一方面跑通了 6DRepNet 摄像头 demo，并记录和分析了真实运行效果；另一方面在 AFLW2000 数据集上复现了论文 Table 1 的定量结果，证明实验流程、权重文件、测试列表和评价逻辑均已与作者设置对齐。
